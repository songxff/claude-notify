## 一、功能概述

当 Claude Code 在后台运行(终端窗口不在前台)时,通过 **Windows 系统通知(Toast)** 提醒用户。

核心能力:
- **三类提醒**:回复完成 / 空闲等待输入 / 需要授权;
- **点击跳转**:点通知横幅或「切换到终端」按钮,一键把终端窗口切回前台;
- **前台静默**:终端正处于前台时不打扰,不弹通知;
- **同类去重**:同一事件的旧通知会被新的替换,不在操作中心堆积;
- **可选远程推送**:可把通知转发到手机(Bark / ntfy / 钉钉 / 自定义 webhook)。

适用环境:Windows 10/11 + 任意终端(当前实测 Warp / `warp-oss.exe`,也兼容 Windows Terminal、pwsh 等)。

---

## 二、触发事件

通知系统挂在 4 个 Claude Code hook 事件上,配置在 `C:\Users\Administrator\.claude\settings.json` 的 `hooks` 段。全部指向同一入口脚本 `claude-notify.ps1`,以参数区分事件。

| Hook 事件 | matcher | 触发时机 | 行为 |
|---|---|---|---|
| `SessionStart` | 无 | 会话开始 | **不弹通知**,只定位终端窗口、按 session_id 缓存句柄 |
| `Stop` | 无 | Claude 回复完成 | 弹 Toast,正文取最后一段回复 |
| `Notification` | `idle_prompt` | 空闲等待用户输入 | 弹 Toast「在等你」 |
| `PermissionRequest` | 无 | 需要工具授权 | 弹 Toast「需要授权」,正文带工具名与参数预览 |

调用形如:
```
powershell -NoProfile -ExecutionPolicy Bypass -File "...\claude-notify.ps1" <事件名>
```
其中 `<事件名>` 为 `SessionStart` / `Stop` / `Notification` / `Permission`。

## 五、核心机制详解

### 5.1 窗口句柄定位与缓存

要"点通知切回终端",必须先知道终端窗口的句柄(HWND)。

- **定位**:`Find-TerminalHwnd` 从当前 PowerShell 进程沿**父进程链**向上走(最多 24 层),取第一个"有可见主窗口且有标题"的祖先进程的 `MainWindowHandle`。冷启动下约 0.8s。
- **缓存**:以 `session_id` 为键,句柄写入 `%TEMP%\ccnotify-<session_id>.hwnd`。`SessionStart` 时定位一次并缓存;后续事件优先读缓存,命中则免去进程枚举。
- **清理**:`SessionStart` 顺手删除 2 天前的旧缓存文件。

### 5.2 前台静默

worker 弹通知前,用 `GetForegroundWindow()` 判断:若当前前台窗口就是终端窗口,说明用户正盯着终端,**直接静默退出**,不打扰。日志记 `terminal foreground -> skip`。

### 5.3 Toast 通知

用 WinRT API(`Windows.UI.Notifications.ToastNotificationManager`)弹系统通知。

- **应用身份**:`AppUserModelId = Claude.Code.Notify`(已在注册表登记,见第六章)。决定通知里显示的应用名。
- **内容**:`ToastGeneric` 模板,两行 `<text>`(标题 + 正文)。
  - `Stop`:标题 `Claude Code · <项目名>`,正文 = 最后一段回复(读 transcript 末尾 400 行,截断 120 字);
  - `Notification`:标题 `Claude Code 在等你 · <项目名>`,正文 = payload 的 message;
  - `Permission`:标题 `Claude Code 需要授权 · <项目名>`,正文 = `请求使用 <工具名> : <参数预览>`(截断 100 字)。
- **去重**:`toast.Tag = 事件名`,`toast.Group = 'claude-code'`。相同 Tag+Group 的旧通知被新的替换。
- **点击行为**:见 5.4。
- **失败兜底**:Toast 弹失败时 `Console.Beep(880,220)` 响一声。

当前 Toast XML 形态:
```xml
<toast launch="claude-jump:focus?hwnd=<HWND>" activationType="protocol">
  <visual><binding template="ToastGeneric">
    <text>标题</text><text>正文</text>
  </binding></visual>
  <actions>
    <action content="切换到终端" activationType="protocol"
            arguments="claude-jump:focus?hwnd=<HWND>"/>
  </actions>
</toast>
```

### 5.4 点击跳转(claude-jump: 协议 + activator)★

**这是历史上最易出问题的环节,务必理解原理。**

- Toast 的 `launch`(点横幅)和 `<action>` 的 `arguments`(点按钮)都设为
  `claude-jump:focus?hwnd=<HWND>`,`activationType="protocol"`。
- 用户点击 → Windows 按注册表 `HKCU\Software\Classes\claude-jump\shell\open\command`
  拉起 `claude-activator.exe "claude-jump:focus?hwnd=<HWND>"`。
- activator 解析出 HWND,把该窗口切到前台,**做完即退出**(不常驻)。

**为什么用"协议拉起新进程"而不是"COM 常驻守护进程":**
Windows 对 `SetForegroundWindow` 有前台锁限制——只有特定进程能真正把任意窗口切到前台。
其中一条例外规则是"**由前台进程启动的进程**":用户点通知时,通知平台(前台)经协议
拉起 activator,这个**全新进程因此持有真正的前台授权**,能做真激活(抬 Z 序、还原最小化)。
而后台常驻的守护进程拿不到这个授权,只能靠 attach 线程的 hack——遇到退化前台时会
"假成功"(API 记账变了、屏幕没反应)。所以采用一次性协议处理器。

**为什么不闪烁:**`claude-activator.exe` 编译为 `winexe`(GUI 子系统),启动**不分配控制台
窗口**,所以没有"控制台一闪"。

**activator 切窗逻辑**(`Focus`):
1. 启动后先 `Sleep(120ms)`,等弹窗横幅收起动画;
2. 关掉前台锁超时(`SPI_SETFOREGROUNDLOCKTIMEOUT`);
3. 最多 60 轮重试(预算约 10 秒),**每轮都直接尝试切换**:opportunistic attach 当前
   前台线程+目标线程(attach 失败也照切)→ 轻点 ALT → `ShowWindow`(最小化则
   `SW_RESTORE`)→ `BringWindowToTop` → `SetForegroundWindow` → `SwitchToThisWindow`;
   切成功(`GetForegroundWindow()==目标`)即返回。每轮约 170ms。
4. 受保护系统 UI(如"操作中心")赖在前台时,`SetForegroundWindow` 自然不生效、目标
   窗口也不会闪;activator 继续轮询,等系统 UI 自行收起后的某一轮即切换成功。所以
   从操作中心面板点击会有约 1~2.5s 延迟;点弹窗横幅基本秒切。
5. **不设"附着不上就纯等"的硬门槛**——曾有版本这么做,导致操作中心占前台超过预算时
   activator 一次都不尝试就 `focus gave up`(见第十章 2026-05-22 第 2 条)。
   节流日志:被挡时每 ~1.5s 记一行 `try#N blocked fg=0x... attach=...`,便于事后诊断。

### 5.5 webhook 远程推送(可选)

worker 末尾按 `notify-config.json` 把通知转发到远程。默认关闭。详见第七章。

## 七、配置:notify-config.json

```json
{
  "webhook": {
    "enabled": false,
    "type": "bark",

    "events": ["Notification", "Permission"]
```

## 八、日志与排错

**日志文件**:`%TEMP%\claude-hooks.log`(即 `C:\Users\Administrator\AppData\Local\Temp\claude-hooks.log`)

日志行前缀:

| 前缀 | 来源 | 典型内容 |
|---|---|---|
| `[entry/<事件>]` | `claude-notify.ps1` | `cached hwnd=...` / `hwnd=... cache=True` |
| `[worker/<事件>]` | `claude-notify-worker.ps1` | `toast shown` / `terminal foreground -> skip` / `toast failed: ...` / `webhook sent` |
| `[activator]` | `claude-activator.exe` | `launched args=...` / `try#N fg-was=... now=...` / `focus OK` / `focus gave up` |

常见排查:

| 现象 | 排查方向 |
|---|---|
| 完全不弹通知 | 看 `[worker]` 是否 `terminal foreground -> skip`;或 `toast failed`;检查 `settings.json` hooks 配置 |
| 点击通知不切窗 | 看 `[activator]` 是否出现。**没出现** = 协议没注册好(查第六章注册表);**出现但 `focus gave up`** = 目标窗口句柄失效或被系统 UI 长期占据 |
| 通知应用名显示成 `Claude.Code.Notify` | AUMID 的 `DisplayName` 丢失,重设第六章注册表 |
| 切窗很慢 | 从"操作中心"面板点击属正常(约 1~1.5s);点弹窗横幅应秒切 |

手动测试协议链路(终端会被切到前台):
```powershell
$h = (Get-Process warp-oss | Select-Object -First 1).MainWindowHandle.ToInt64()
Start-Process "claude-jump:focus?hwnd=$h"
```

## 九、维护:重新编译 activator

改了 `claude-activator.cs` 后必须重新编译。`claude-activator.exe` 是一次性进程、不常驻,
正常情况下编译时不会被占用;若曾用旧 COM 版本、有残留进程,先杀掉再编。

```powershell
# (可选)杀掉可能残留的进程
Get-Process claude-activator -ErrorAction SilentlyContinue | Stop-Process -Force

# 编译(winexe = 无控制台窗口 = 点击无闪烁)
$csc = "$env:WINDIR\Microsoft.NET\Framework64\v4.0.30319\csc.exe"
& $csc /nologo /target:winexe `
    /out:"C:\Users\Administrator\.claude\hooks\claude-activator.exe" `
    "C:\Users\Administrator\.claude\hooks\claude-activator.cs"
```

> `/target:winexe` 不可漏——若编成 `exe`(控制台子系统),点击通知会闪一下黑窗。
> 校验:exe 的 PE Subsystem 应为 `2`(GUI)。

## 十、更新记录

> 每次改动通知系统,在此**追加**一条(最新在上)。格式:日期 · 改动 · 原因。

### 2026-05-22 · 修复 activator 在操作中心占前台时直接放弃切窗

**现象**:点击通知有时不跳转,日志只有 `launched` 紧跟 `focus gave up`,中间无任何尝试记录。

**根因**:`claude-activator.cs` 的 `Focus` 循环里有一道硬门槛——
`if (!AttachThreadInput(...)) { continue; }`:附着不上当前前台线程(例如前台是受保护的
"操作中心")时**纯等、一轮都不尝试切窗**。而循环预算仅 50 轮 × 70ms ≈ 3.5 秒。
当从操作中心面板点击、操作中心赖在前台超过 3.5 秒时,activator 一次都没试就放弃了。
(协议拉起的 activator 本就持有真前台授权,根本不需要靠 attach,这道硬门槛是从早期
COM 守护进程方案误带过来的——COM 后台进程没授权才必须靠 attach。)

**改动**(`claude-activator.cs` 的 `Focus`):
- 去掉硬门槛,**每轮都直接尝试切换**,attach 改为 opportunistic(成功则用、失败照切);
- 预算由 50 轮放宽到 60 轮(约 10 秒),足够等操作中心退场;
- 加节流诊断日志:被挡时每 ~1.5s 记一行 `try#N blocked fg=0x... attach=...`。

**效果**:操作中心等受保护 UI 占前台时不再提前放弃,会一直轮询到它收起后切换成功。

### 2026-05-22 · 由 COM 守护进程改回 claude-jump: 协议处理器

**背景**:此前为消除点击通知时的"控制台一闪",曾把激活机制从协议处理器改成
COM 常驻守护进程(`INotificationActivationCallback` + `CoRegisterClassObject`)。
结果点击通知**完全打不开终端**——后台守护进程拿不到 `SetForegroundWindow` 真授权,
attach 线程的 hack 只改了 API 记账,日志写 `focus OK` 实为假阳性,屏幕无反应。

**改动**:
- `claude-activator.cs`:删除全部 COM 脚手架(CLSID、类厂、`INotificationActivationCallback`、
  单实例 Mutex、12 小时常驻),改为**一次性协议处理器**——从命令行参数解析
  `focus?hwnd=N`,切窗,退出。轮询间隔调紧(每轮约 170ms,最多 40 轮)。
- 注册表:新增 `claude-jump:` 协议(`shell\open\command` → activator);
  删除旧的 `CLSID\{2F3E9C1A-...}` 键与 `AppUserModelId\...\CustomActivator` 值;
  给 AUMID 补 `DisplayName = Claude Code`。
- `claude-notify-worker.ps1`:Toast 的 `launch` / `<action> arguments` 改为
  `claude-jump:focus?hwnd=N`,`activationType` 由 `foreground` 改 `protocol`;
  删除"确保 COM 守护进程在运行"那段。
- `claude-notify.ps1`:删除 `SessionStart` 里启动 COM 守护进程那段。

**效果**:点击通知可靠切回终端(协议拉起的新进程持有真前台授权);`winexe` 编译
保证无闪烁;代码大幅简化。从"操作中心"面板点击约 1~1.5s,点弹窗横幅秒切。

### (更早) · COM 守护进程方案

为消除协议处理器的控制台闪烁,改用 COM 常驻守护进程接管 Toast 点击。**已废弃**
(见上一条):后台进程无法可靠激活窗口。

### (最初) · claude-jump:// 协议方案

最初版本即用自定义协议处理器激活窗口,可靠但点击时控制台会闪一下。
